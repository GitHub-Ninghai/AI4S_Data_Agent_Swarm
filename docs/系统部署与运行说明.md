# AI4S Data Agent Swarm 系统部署与运行说明

## 1. 文档目的

本文档用于支持组委会对 `AI4S Data Agent Swarm` 的部署、运行、复现与接口验证。本文覆盖：

- 系统架构与组件说明
- 运行环境要求
- Docker 打包与启动方式
- 基础验证与测试方法
- 日志、数据目录与排障方式
- 可供验证的 REST / WebSocket 接口说明

当前仓库已提供可直接构建的 Docker 交付物：

- `Dockerfile`
- `docker-compose.yml`

默认对外服务地址：

- Web UI: `http://<部署主机>:3456/`
- Health Check: `http://<部署主机>:3456/api/health`
- WebSocket: `ws://<部署主机>:3456/ws`

## 2. 系统架构

系统为单体 Web 服务，运行时包含以下部分：

1. `server/`
   Express + WebSocket 后端，负责项目、Agent、Task、事件流、文件上传、Copilot 会话与数据流水线接口。
2. `web/`
   React + Vite 前端。生产模式下先构建静态资源，再由后端统一托管。
3. `data/`
   本地 JSON/JSONL 持久化目录，无需数据库。
4. `/workspace`
   容器内项目工作目录挂载点。评测时在 UI / API 中创建项目，`path` 字段应填写容器内绝对路径，例如 `/workspace`。

生产部署形态：

```text
Browser
  -> HTTP / WS :3456
Express Server
  -> REST API (/api/*)
  -> Hook API (/event)
  -> WebSocket (/ws)
  -> Static Web Assets (web/dist)
  -> File Stores (data/*.json)
  -> Event Logs (data/events/*.jsonl, *.jsonl.gz)
  -> Hook Logs (data/logs/hooks.log)
  -> Project Workspace (/workspace)
```

## 3. 运行环境要求

### 3.1 操作系统

- Linux x86_64，推荐 Ubuntu 22.04 / Debian 12
- macOS / Windows 可通过 Docker Desktop 运行

### 3.2 基础软件

- Docker Engine 24+ 或 Docker Desktop 4+
- Docker Compose v2

如果采用源码方式运行，还需要：

- Node.js >= 18
- npm >= 9

### 3.3 硬件资源建议

最低建议：

- CPU: 2 核
- 内存: 4 GB
- 磁盘: 10 GB 可用空间

推荐配置：

- CPU: 4 核及以上
- 内存: 8 GB 及以上
- 磁盘: 20 GB 及以上

说明：

- 基础 CRUD、UI、日志和健康检查验证对资源要求较低。
- 若实际执行 Agent 任务、Copilot 能力或 PDF 解析任务，还会额外依赖外部模型/API，并增加 CPU、内存与网络开销。

### 3.4 环境变量

核心变量如下：

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `PORT` | `3456` | 服务端口 |
| `HOST` | `127.0.0.1` / Docker 中为 `0.0.0.0` | 服务监听地址 |
| `MAX_CONCURRENT_TASKS` | `10` | 最大并发任务数 |
| `MAX_WS_CLIENTS` | `10` | WebSocket 最大连接数 |
| `TOOL_APPROVAL_TIMEOUT_MS` | `300000` | 工具审批超时 |
| `USER_MESSAGE_TIMEOUT_MS` | `1800000` | 用户消息等待超时 |

可选变量如下：

| 变量 | 说明 |
| --- | --- |
| `ANTHROPIC_AUTH_TOKEN` | Anthropic API Token（与 API_KEY 二选一） |
| `ANTHROPIC_API_KEY` | 第三方模型 API Key（如 DeepSeek、OpenAI） |
| `ANTHROPIC_BASE_URL` | 模型服务基地址 |
| `ANTHROPIC_MODEL` | 模型名（如 `deepseek-chat`） |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | 备选模型名 |
| `COPILOT_MODEL` | Copilot 模型名，默认 `glm-5` |
| `API_TIMEOUT_MS` | API 调用超时（毫秒），默认 `600000` |
| `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` | 设为 `1` 可减少 Claude Code 非必要网络流量 |
| `MINERU_TOKEN` | MinerU Open API Token（PDF 解析） |
| `MINERU_API_KEY` | MinerU API Key（与 TOKEN 等价） |

## 4. Docker 打包与启动

### 4.1 构建镜像

在仓库根目录执行：

```bash
docker build -t ai4s-data-agent-swarm:latest .
```

如部署环境需通过国内镜像加速 npm 安装，可使用构建参数：

```bash
docker build -t ai4s-data-agent-swarm:latest \
  --build-arg NPM_REGISTRY=https://registry.npmmirror.com .
```

构建说明：

- 镜像使用 Node.js 20 (bookworm-slim) 多阶段构建，包含 TypeScript 编译和前端 Vite 构建
- 默认使用 npm 官方 registry；国内用户可通过 `--build-arg NPM_REGISTRY` 指定镜像源
- Web 前端构建跳过 `tsc -b` 类型检查（预存 TS 类型错误不阻塞构建），类型检查在独立 CI 步骤中处理
- 构建产物包括：`server/dist/`（后端编译结果）、`web/dist/`（前端构建结果）

### 4.2 使用 Docker Compose 启动

```bash
cp .env.example .env
mkdir -p workspace
docker compose up --build -d
```

启动后访问：

```text
http://localhost:3456
```

查看运行状态：

```bash
docker compose ps
docker compose logs -f
```

停止服务：

```bash
docker compose down
```

### 4.3 挂载目录说明

`docker-compose.yml` 默认挂载：

| 宿主机目录 | 容器内目录 | 作用 |
| --- | --- | --- |
| `./data` | `/app/data` | 系统持久化数据、事件流、日志 |
| `./workspace` | `/workspace` | 项目工作目录、上传文件、论文文件等 |

注意：

- 项目创建接口会校验路径必须为容器内绝对路径。
- 评测时建议直接使用 `/workspace` 作为项目路径。

## 5. 运行验证方法

### 5.1 基础可用性验证

1. 打开 `http://localhost:3456`
2. 检查页面是否正常加载
3. 调用健康检查接口：

```bash
curl http://localhost:3456/api/health
```

预期返回示例：

```json
{
  "status": "ok",
  "version": "0.1.0",
  "uptime": 12.345,
  "activeTaskCount": 0,
  "maxConcurrentTasks": 10,
  "storageOk": true
}
```

### 5.2 项目创建验证

先确保挂载目录存在：

```bash
mkdir -p workspace
```

再创建项目：

```bash
curl -X POST http://localhost:3456/api/projects \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "eval-project",
    "path": "/workspace",
    "description": "committee verification project"
  }'
```

### 5.3 自动化测试验证

如果在源码环境验证：

```bash
# 后端测试（需先在 server/ 目录执行 npm install）
cd server && npx vitest run

# 注意：Web 前端包含预存 TS 类型错误，直接 npx vitest run 可能不通过
# 建议仅对核心接口进行 HTTP 验证
```

如果只验证 Docker 交付物，建议至少执行：

```bash
# 1. 健康检查
curl http://localhost:3456/api/health

# 2. 登录后获取业务接口数据
TOKEN=$(curl -s -X POST http://localhost:3456/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"account":"admin","password":"admin123"}' | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['token'])")

# 3. 验证业务接口
curl http://localhost:3456/api/projects -H "Authorization: Bearer $TOKEN"
curl http://localhost:3456/api/agents -H "Authorization: Bearer $TOKEN"
```

## 7. 对外服务与接口说明

### 7.1 认证方式

当前服务对本地/容器内验证接口默认不启用额外鉴权。

- HTTP API：无鉴权
- WebSocket：无鉴权

说明：

- 为便于组委会复现与验证，当前版本优先提供开箱可测的本地部署能力。
- 若后续需要公网发布，建议在反向代理层增加鉴权、IP 白名单或网关认证。

### 7.2 返回格式

成功返回通常为 JSON 对象，例如：

```json
{ "ok": true }
```

或：

```json
{ "projects": [] }
```

失败返回统一格式：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "path is required"
  }
}
```

### 7.3 认证说明

除健康检查 `/api/health` 和 Hook 接口 `/event` 外，所有业务接口均需 JWT 认证。

获取 Token：

```bash
curl -X POST http://localhost:3456/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"account":"admin","password":"admin123"}'
```

成功返回：

```json
{
  "code": 0,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "user": { "id": "...", "name": "指挥员", "email": "admin", "role": "admin" }
  },
  "message": "ok"
}
```

后续接口请求需在 Header 中添加：

```
Authorization: Bearer <token>
```

默认管理员账号：`admin` / `admin123`（首次启动时自动创建）。

### 7.4 核心接口

#### 7.4.1 健康检查

- 方法：`GET`
- 路径：`/api/health`
- 用途：检查服务是否存活、返回版本和当前任务状态

#### 7.4.2 项目管理

1. `GET /api/projects`
   返回项目列表。
2. `POST /api/projects`
   创建项目。

请求体示例：

```json
{
  "name": "eval-project",
  "path": "/workspace",
  "description": "committee verification project"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name` | string | 是 | 仅允许 `[a-zA-Z0-9_-]` |
| `path` | string | 是 | 容器内绝对路径，且必须已存在 |
| `description` | string | 否 | 项目说明 |

3. `PUT /api/projects/:id`
   更新项目信息。
4. `DELETE /api/projects/:id`
   删除项目。若存在 `Running` 或 `Stuck` 任务会返回冲突错误。

#### 7.4.3 Agent 管理

1. `GET /api/agents`
2. `GET /api/agents/:id`
3. `POST /api/agents`
4. `PUT /api/agents/:id`
5. `GET /api/agents/:id/stats`
6. `DELETE /api/agents/:id`

创建 Agent 请求体示例：

```json
{
  "name": "验证用Agent",
  "avatar": "🤖",
  "role": "Verification Agent",
  "prompt": "You are a verification agent responsible for evaluation tasks.",
  "projectId": "project-id",
  "maxTurns": 50,
  "maxBudgetUsd": 2
}
```

#### 7.4.4 Task 管理

1. `GET /api/tasks`
   支持 `projectId`、`status`、`agentId`、`q`、`page`、`limit`、`includeDeleted` 查询参数。
2. `GET /api/tasks/:id`
3. `GET /api/tasks/:id/events`
   支持 `page`、`limit`、`type`。
4. `GET /api/tasks/:id/sdk-status`
5. `POST /api/tasks`
6. `PUT /api/tasks/:id`
7. `DELETE /api/tasks/:id`
8. `POST /api/tasks/:id/start`
9. `POST /api/tasks/:id/stop`
10. `POST /api/tasks/:id/done`
11. `POST /api/tasks/:id/message`
12. `POST /api/tasks/:id/approve-tool`
13. `POST /api/tasks/:id/retry`

创建 Task 请求体示例：

```json
{
  "title": "验证任务",
  "description": "使用当前项目进行最小化验证。",
  "agentId": "agent-id",
  "projectId": "project-id",
  "priority": 1,
  "tags": ["eval"],
  "maxTurns": 20,
  "maxBudgetUsd": 1
}
```

#### 7.4.5 文件上传

1. `POST /api/files/upload`
   `multipart/form-data` 上传 PDF，字段：
   - `projectId`: 所属项目 ID
   - `files`: 最多 20 个 PDF
2. `GET /api/files/:projectId`
3. `DELETE /api/files/:projectId/:fileId`

#### 7.4.6 数据流水线

1. `POST /api/pipeline/create`

请求体示例：

```json
{
  "pipelineType": "qa",
  "projectId": "project-id",
  "pdfFiles": ["uploads/sample.pdf"],
  "maxTurns": 50,
  "maxBudgetUsd": 3
}
```

`pipelineType` 当前支持：

- `qa`
- `scievo`

#### 7.4.7 Copilot

1. `POST /api/copilot/session`
2. `DELETE /api/copilot/session/:id`
3. `POST /api/copilot/chat`
4. `POST /api/copilot/confirm`

`POST /api/copilot/chat` 请求体示例：

```json
{
  "sessionId": "optional-session-id",
  "message": "帮我创建一个验证任务"
}
```

#### 7.4.8 Hook 事件接口

- 方法：`POST`
- 路径：`/event`
- 用途：接收外部 Hook 事件并写入任务事件流/日志

请求体示例：

```json
{
  "hook_event_name": "PreToolUse",
  "session_id": "sdk-session-id",
  "tool_name": "Read",
  "tool_input": "README.md"
}
```

#### 7.4.9 WebSocket

- 路径：`/ws`
- 协议：WebSocket
- 认证：无

典型消息类型包括：

- `task:update`
- `agent:update`
- `event:new`
- `tool:approval`
- `task:budget`
- `notification`
- `error`

## 8. 组委会建议验证流程

建议按以下顺序验证：

1. `cp .env.example .env && mkdir -p workspace`
2. `docker compose up --build -d`
3. 访问 `http://localhost:3456` — 确认前端页面正常加载
4. `curl http://localhost:3456/api/health` — 确认服务返回 `{"status":"ok"}`
5. 登录获取 Token：
   ```bash
   TOKEN=$(curl -s -X POST http://localhost:3456/api/auth/login \
     -H 'Content-Type: application/json' \
     -d '{"account":"admin","password":"admin123"}' | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['token'])")
   ```
6. `POST /api/projects` 创建项目，`path` 使用 `/workspace`：
   ```bash
   curl -s -X POST http://localhost:3456/api/projects \
     -H 'Content-Type: application/json' \
     -H "Authorization: Bearer $TOKEN" \
     -d '{"name":"eval-project","path":"/workspace","description":"committee verification project"}'
   ```
7. `GET /api/projects`、`GET /api/agents`、`GET /api/tasks` 验证基础接口（均需携带 Token）
8. 查看 `docker compose logs -f` 与 `data/` 目录，确认日志与持久化正常

## 9. 已知说明

1. 当前文档交付的是“可本地复现、可容器化验证”的服务，不包含公网托管地址。
2. 若需互联网可访问版本，可在当前镜像基础上通过云主机或反向代理发布，服务入口仍为 `:3456`。
3. Agent/Copilot/PDF 解析等高级能力依赖外部模型或第三方服务凭据；未配置凭据时，基础界面、健康检查、项目/Agent/Task 管理与日志功能仍可验证。
4. Web 前端存在预存 TypeScript 类型错误，不影响运行时功能。Docker 构建跳过 `tsc -b` 类型检查阶段直接使用 Vite 构建。若在源码环境下运行，可通过 `cd web && npx vite build` 绕过类型检查构建。完整的类型修复将在后续迭代中处理。
